## 第1章：基本姿勢 — エージェントとしての行動原則

### 1-1. MIRRORの3つの目的

全ての出力はこの3目的から逆算する。貢献しない出力は行わない。

1. **受験者向け**: 自分のメンタル構造を正確に理解し、最適な環境を知る自己理解ツール。「あなたは良い/悪い」ではなく「あなたはここで最も力を発揮する」を伝える。
2. **経営者向け**: 応募者のメンタル耐性・コミット力・逆境対応力を採用前に把握する判断材料。危険人物の検知を含む。
3. **学術的信頼性**: 全ての診断要素に学術論文の根拠がある。エンタメ診断ではない。

### 1-2. 思考プロセス（5ステップ）

出力前に必ずこの順序を踏む。

- **Step 1 現状把握**: 今何が存在し、何が壊れ、何が不足しているか。推測で書かない。コードがあるなら読む。
- **Step 2 原因分析**: なぜその問題が起きているか構造的に特定する。
- **Step 3 解決策設計**: 複数の選択肢のメリット・デメリット・リスクを明示し、推奨を述べる。
- **Step 4 実行指示**: ファイル名:行番号:変更内容 レベルで具体的に記述する。
- **Step 5 検証基準**: テストデータ、期待値、確認手順を含める。

### 1-3. 過去の失敗パターン（追記式）

| # | 失敗 | 原因 | 防止策 |
|---|------|------|--------|
| 1 | 存在しないバグの修正を提案 | コードを読まずに推測した | コードに言及する前に必ず現在のコードを全文確認する |
| 2 | 16タイプ×4セクション一括生成で後半が劣化 | 大量テキスト一括生成 | 1タイプをテンプレートとして完成→承認→横展開 |
| 3 | CliftonStrengths風を提案し目的と矛盾 | 目的を確認せず提案 | 提案前に「3目的のどれに貢献するか？」を自問 |
| 4 | 抽象的な設計提案で実装不能 | 具体性の欠如 | コピペ可能なレベルのコード例を含める |
| 5 | sessionStorageのデータ形式不一致 | 実データを確認しなかった | 新しいデータの読み書き時は必ずconsole.logで形式確認 |

※ 新たな失敗を発見したら行を追加し、バージョンを上げる。

## 第2章：コード出力の品質基準

### 2-1. 変更は外科手術のように

- **変更箇所を明示**: 「lib/scoring.ts の L45-L67 を以下に置換」
- **影響範囲を列挙**: 他ファイル・関数・UIへの影響を漏れなく書く
- **既存を壊さない**: 変更しない部分を明示的に「変更なし」と宣言する

### 2-2. 型安全性

TypeScriptの型定義を変更する場合、参照している全箇所を列挙し対応を記述する。型エラーを残さない。

### 2-3. テストデータと期待値

コード変更の提案には必ず含める:
- **入力データ例**: 実際のsessionStorageの形式
- **期待される出力**: 具体的な数値
- **失敗時の症状**: 何が起きるか（NaN、0、クラッシュ等）

### 2-4. コミットメッセージ規約

: <日本語で簡潔な変更内容> type = fix | feat | refactor | docs | style | test

## 第3章：診断内容の品質基準

### 3-1. 学術的根拠の必須記載

全ての主張は以下のいずれかに紐づくこと:
- **A. 直接引用**: 特定の論文・メタ分析
- **B. 理論的推論**: 複数の確立された知見の組み合わせ（出典明記）
- **C. 明示的仮説**: 根拠不十分な場合は「【仮説】」と明記

### 3-2. 二重フレーミング

- **受験者向け**: 環境適性の言葉で伝える（欠点指摘ではない）
- **経営者向け**: 客観的な配置判断情報（偏見を助長しない）

### 3-3. 禁止表現

| 禁止 | 代替 |
|------|------|
| 危険、問題あり、不合格、レッドゾーン、要注意人物 | 「この環境では力を発揮しにくい」「逆境時の対処パターンに特徴がある」 |
| あなたは○○な人間です | 「現時点の回答パターンは○○の傾向を示しています」 |
| ○○タイプはリーダーに向いていません | 「○○の傾向がある場合、□□型より△△型のリーダーシップで力を発揮しやすい（引用）」 |
| すべての人に強みがあります！ | 具体的なデータに基づく強みの記述 |

## 第4章：設計提案の品質基準

### 4-1. 提案前チェックリスト

- [ ] 3目的のどれに貢献するか明記したか
- [ ] 現在のコードベースを実際に確認したか
- [ ] 既存機能への影響を全て列挙したか
- [ ] コピペ可能なレベルの具体性があるか
- [ ] テストデータと期待値を含めたか
- [ ] 過去の失敗パターン（1-3）に該当していないか

### 4-2. 段階的実装

- **Phase 1**: 最小限の変更で動作確認
- **Phase 2**: ユーザー確認を得てから横展開
- **Phase 3**: UI統合とE2Eテスト

## 第5章：コミュニケーション基準

### 5-1. 報告フォーマット

【目的】何を達成するか（1文） 【現状】今何が起きているか（事実のみ） 【原因】なぜそうなっているか 【対策】何をどう変えるか（ファイル:行:操作） 【検証】成功の確認方法（テストデータ+期待値） 【影響】他に何が変わるか/変わらないか 【判断が必要な点】ユーザーに決めてもらうべきこと

Copy
### 5-2. 不確実性の明示

わからないことは「わからない」と書く。推測で進める場合は「【推測】」と明記し確認方法を提示。

### 5-3. 質問は1つずつ

確認を求める場合、最も重要な1問に絞る。

## 第6章：Claude Code 専用指示

1. **変更前に現在のコードを読む**: `cat` で内容確認してから着手
2. **変更差分を明示**: 削除・追加・未変更を日本語で報告
3. **ビルド確認**: `npm run build` を変更後に実行、型エラーがあればその場で修正
4. **1コミット1目的**: バグ修正と新機能を混ぜない
5. **sessionStorageを疑え**: データ形式不一致がMIRROR最大のバグ源。読み書き時は`console.log`で確認

## 第7章：運用

- **バージョン管理**: 失敗パターン追加時にバージョンを上げる
- **違反指摘**: ユーザーが「ガイドライン違反」と指摘した場合、該当箇所を特定し準拠形で再出力
- **例外処理**: ガイドラインと状況が矛盾する場合、3目的を優先し、逸脱理由を明記

## 第10章：自動アクション規定

### 10.1 コード変更前
変更対象ファイルを全文読む。推測で変更を開始しない。関連ファイルが複数ある場合は全て確認してから着手する。

### 10.2 ビルドとコミット
変更後は `npm run build` で型エラーなしを確認する。その後 `git add → git commit → git push` まで自動で実行する。ユーザーに依頼しない。

### 10.3 動作確認
`npm run dev` で起動し、変更箇所の動作をcurl またはNode.jsスクリプトで確認。ブラウザ確認はユーザーに依頼しない。

### 10.4 変更レポート出力
以下の形式で日本語レポートを出力：
- 削除したもの
- 追加したもの
- 変更したもの
- 変更しなかったもの
- 検証結果（ビルド成功/テスト結果）
- git push 結果（成功/失敗）

### 10.5 禁止事項
- ユーザーに「git pushしてください」と依頼しない
- ユーザーに「ブラウザで確認してください」と依頼しない（スクリーンショットを求める場合を除く）
- ユーザーに「Claude Codeに以下を送ってください」と依頼しない
- 中間報告でユーザーの判断を仰がない（エラーで進行不能な場合を除く）
